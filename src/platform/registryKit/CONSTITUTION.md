# Platform Constitution — Registry Layer

Every registry in the platform MUST satisfy the following. New
registries pass this checklist as their acceptance criteria.

## Checklist

- [ ] **Schema validation** — id, semver, name, description, category
      enforced at register.
- [ ] **Metadata validation** — required RegistrationBase fields
      present, non-empty.
- [ ] **Duplicate detection** — same id twice throws; never
      silently overwrites.
- [ ] **Dependency validation** — cross-registry references validated
      at register time when possible; deferred to install when not
      (documented per-registry).
- [ ] **Version validation** — strict semver required.
- [ ] **Deprecation support** — `deprecation.deprecatedSince` +
      `replacedBy` fields; `selfCheck()` warns on dangling refs.
- [ ] **Search** — weighted-keyword `.search(query, limit?)` matching
      id, name, keywords, tags, description, category.
- [ ] **Filtering** — `.listByCategory()`, `.listByTag()`, and any
      domain secondary indexes.
- [ ] **Sorting** — `.list()` returns registration-order; callers may
      sort. (No enforced sort — collapse would over-index on one
      caller's preference.)
- [ ] **Tags** — cross-cutting filter facets, first-class on
      `RegistrationBase`.
- [ ] **Categories** — primary index key on every registry.
- [ ] **AI metadata** — `.describe(id)` returns a one-line
      AI-consumable summary; extensible via `describeRegistration()`
      `extraFacts`.
- [ ] **Telemetry hooks** — `onRegister`, `onGet`, `onMiss` optional
      hooks fire fire-and-forget.
- [ ] **Analytics hooks** — `analytics.onEvent` fires business events
      (added / searched / hydrated).
- [ ] **Future DB compatibility** — `.snapshot()` returns a JSON-safe
      dump; factory `seed` option accepts hydration from a snapshot.

## Non-negotiables

1. **No hand-rolled registries.** Every registry uses
   `createRegistry` from `@/platform/registryKit`. Duplicate
   implementations are architectural drift and must be refactored
   before merge.

2. **Manifests are frozen.** After `.register()` returns, the
   registration is immutable. No mutation of `.get()` results.

3. **Aliases are transparent.** `.get(alias)` returns the canonical
   registration; callers never branch on whether they used an alias.

4. **Registrations are pure data + one component reference.** No
   I/O in a manifest. No lifecycle hooks. Renderers are React
   components consumed at render time.

5. **Cross-registry references are validated at register time when
   feasible.** When a validation cost is prohibitive (large graph
   walks), defer to install time and document the deferral in the
   registry file header.

## Grandfathered exceptions

- **`aiGateway`** — not a content registry. Manages LLM provider
  adapters, which is a different architectural concern (routing +
  health + budget + failover). It will move to a future *AI Provider
  Manager* subsystem in a later milestone. It does not use
  `registryKit`, deliberately.

- **`ProcessorRegistry` (payment)** — migrated in Milestone 1 behind
  a facade that derives `RegistrationBase` fields from `providerId`.
  Enriched processor manifests are optional; when authored, they
  unlock marketplace metadata + proper search + describe. The 20+
  existing processor files did not need to change.

## Amendments

To add or change a constitution item:

1. Update this document.
2. Update `registryKit/types.ts` if the change is structural.
3. Add tests to `__tests__/`.
4. Retrofit or verify every existing registry.

Amendments should be additive whenever possible; breaking changes to
`RegistrationBase` bump the kit's minor version and require a
migration.

## Amendment 1 — M2 Architectural Principle (2026-07-05)

The Design System is now the foundation of the entire platform.

- **Everything generated by AI must come from the Design System.**
- **AI must never invent layouts.**
- **AI must compose layouts** by selecting registered building blocks.

### Containers

- Containers become the layout owners.
- Sections never control positioning.
- Containers control: `width`, `spacing`, `alignment`, `stacking`,
  `responsive behaviour`, `overlays`, `sidebars`, `grids`, `masonry`,
  `floating elements`, `dashboards`, `mobile arrangements`.
- Sections contain content only.

### Design Registry

- The Design Registry is the **single source of truth** for every
  reusable UI primitive.
- Every primitive must expose machine-readable metadata including:
  `id`, `category`, `tags`, `dependencies`, `compatibleContainers`,
  `compatibleThemes`, `supportedDevices`, `animationSupport`,
  `accessibilityStatus`, `performanceCost`, `searchKeywords`,
  `version`, `deprecation`.
- The AI Builder discovers components ONLY through this registry.

### Theme Extension (foundational — never defer)

Themes must be data-driven, not hard-coded. Every theme controls:

- colour palettes
- typography
- spacing
- border radius
- elevation
- shadows
- gradients
- icon styles
- animation presets
- dark mode
- brand overrides

### Design Tokens (foundational — never defer)

The token system is the backbone of the Design System. Every
component consumes tokens. Never hard-code values. Tokens cover:

- colours
- typography
- spacing
- sizing
- radius
- borders
- shadows
- opacity
- motion
- transitions
- z-index
- icons
- illustrations
- images
- breakpoints

Future AI generates entire themes by swapping token sets.

### Future AI pipeline

The AI composes entirely from registries rather than generating
arbitrary UI:

```
Prompt
  ↓
Industry Detection
  ↓
Business Detection
  ↓
Theme Selection
  ↓
Token Set Selection
  ↓
Container Selection
  ↓
Layout Composition
  ↓
Section Selection
  ↓
Component Selection
  ↓
Asset Selection
  ↓
Animation Selection
  ↓
App Recommendation
  ↓
Final Responsive App
```

### Development strategy — AI intelligence layers

Post-M2, work is organised as intelligence layers rather than
features:

1. **Foundation Layer** — Registries, design system, tokens, themes.
2. **Composition Layer** — AI chooses and assembles layouts from the
   foundation.
3. **Content Layer** — Images, icons, copy, videos, industry assets.
4. **Marketplace Layer** — Apps, Packs, Templates, Plugins.
5. **Experience Layer** — Editing, drag-and-drop, personalization,
   publishing.

This is the platform's structural direction from Milestone 3 onward.

### Milestone sequencing

Milestone sequencing from M3 onward is governed by the **Master
Platform Roadmap** at `ROADMAP.md`. Every session's implementation
plan must cite its milestone + batch. Do not begin a milestone until
the previous milestone's completion report is approved.

## Amendment 2 — Business Operating System (2026-07-05)

The platform is not a website builder. It is a **Business Operating
System generator**.

Every generated app is assembled from a **stack of layers**, each
independently versioned, discoverable, replaceable, and marketplace-
ready:

```
Business
  ↓
Brand
  ↓
Theme
  ↓
Design Tokens
  ↓
Navigation
  ↓
Layouts
  ↓
Containers
  ↓
Sections
  ↓
Components
  ↓
Business Apps
  ↓
CRM
  ↓
Bookings
  ↓
Payments
  ↓
Analytics
  ↓
SEO
  ↓
Automation
  ↓
Publishing
```

### Implications

- **Loveable ceiling**: interface generation. Ours: full business
  operating system tailored to a specific business.
- **Every layer must be registry-driven** — no hand-coded layers.
- **AI composition respects the layer order** — top layers constrain
  the composition space of lower layers (a Business chooses a Brand;
  a Brand chooses a Theme; a Theme constrains valid Token sets, etc.).
- **Sessions from this point onward** operate on the question "what
  *layer* are we completing next?" rather than "what *feature* are we
  building?"
- **Milestone naming** post-M3 must reference the layer they advance
  (e.g. "M4 — Component & Section Layer Depth" not "M4 — Add More
  Sections").

| Publishing | existing publish pipeline (`/api/studio/publish-live`) |

Every layer above the horizontal cut line (Containers) is a
composition input; every layer below is a runtime concern.

## Amendment 3 — Chief Platform Architect operating framework (2026-07-05)

From this point onward the platform is governed as an **engineering
operating system**, not an app builder. The engineer's role is
permanent Chief Platform Architect. Every batch, milestone, and
proposal is measured against this framework — not against feature
completion.

### The 5-phase process (mandatory per batch)

**Phase 1 — Architecture Review** *(no code)*
Before writing any code for a batch, produce an **Architecture
Review Report** covering:
- Constitution + Roadmap re-read
- Every affected registry re-audited
- Every affected system re-audited
- Improvements identified
- Duplication identified
- Technical debt identified
- Future scaling issues identified
- AI limitations identified
- Performance risks identified
- Developer experience improvements identified
- User experience improvements identified
- Opportunities to simplify future milestones identified

**Phase 2 — Platform Improvement Proposal** *(no code; only if
Phase 1 discovered a superior design)*
If Phase 1 discovers a superior architecture, **stop** and produce
a Platform Improvement Proposal covering:
- Why it is better
- Long-term impact
- Migration strategy
- Risk assessment
- Whether existing code changes
- Whether APIs remain compatible
- Whether AI becomes more deterministic
- Whether performance improves
- Whether maintainability improves

Wait for approval before continuing.

**Phase 3 — Batch Planning** *(after Phase 2 approval)*
Produce the batch implementation plan. Every batch entry must
include:
- Purpose
- Files changed
- Registries affected
- Performance impact
- AI impact
- Developer impact
- Backward compatibility
- Risk level
- Testing strategy
- Exit criteria
- Checkpoint

**Phase 4 — Implementation**
Only now begin coding. Every batch must finish with:
- Typecheck (0 new errors)
- Lint (0 new warnings)
- Registry validation (every registry `.selfCheck()` clean)
- Performance check (bundle-size delta reported)
- Accessibility check (Radix baseline verified)
- SelfCheck (cross-registry sanity)
- Constitution compliance (checklist per §Amendment 1)

**Phase 5 — Batch Report**
Every completed batch produces:
- Summary
- Architecture changes
- Technical debt removed
- Future debt created
- Performance metrics
- Registry metrics
- Tests passed
- Files changed
- Updated PMM scores (see below)
- Next recommendation

### The 4 rules

**Continuous Improvement Rule.** The architect is *authorised* to
improve the platform architecture whenever a superior design is
discovered. The architect is *not authorised* to silently change
architecture. On discovery: stop, explain, justify, estimate
impact, request approval — then implement.

**AI Quality Rule.** Before implementing any AI feature, ask:
- Can this be deterministic instead?
- Can registry metadata replace AI?
- Can configuration replace prompting?
- Can composition replace generation?

Only use LLM reasoning when deterministic systems cannot solve the
problem.

**Scalability Rule.** Every implementation must remain capable of
scaling to:
- 500+ Apps
- 2,000+ Sections
- 500+ Containers
- 250+ Layouts
- 100+ Themes
- 50,000+ Assets
- 100,000+ Businesses
- Millions of generated pages

No implementation may assume small datasets. Every registry read,
every AI selection, every layout render must remain O(log n) or O(1)
against the above scale targets.

**Loveable++ Rule.** The goal is no longer Loveable parity. The
platform must exceed Loveable in every measurable engineering
category: Architecture, AI determinism, Registry design, Theme
engine, Token system, Marketplace, Extensibility, White-label
support, Multi-tenant support, Plugin ecosystem, Performance,
Accessibility, Testing, Maintainability, Enterprise readiness.
Visual polish must eventually equal or exceed Loveable while
retaining deterministic architecture.

### Platform Maturity Model — the reporting standard

The **Platform Maturity Model (PMM)** at `PLATFORM_MATURITY_MODEL.md`
is the objective scorecard. Every batch report updates PMM scores.
Every milestone completion is measured against PMM targets. "M3 is
complete" is *not* an acceptable claim; "M3 raised Foundation from
82% → 95%, AI Composition from 45% → 68%, Business OS from 50% →
72%" is.

**Score bands:**
- 0–20% — placeholder / stub / not started
- 21–50% — foundational infrastructure in place, not production-ready
- 51–75% — production-usable for the common path, gaps at edges
- 76–90% — production-ready with known deferred work
- 91–100% — production-hardened with tests + monitoring

### Final Success Definition

The platform is only considered complete when it can:

1. Generate complete business platforms from natural language.
2. Generate websites, mobile apps, customer portals, admin
   dashboards, booking systems, CRMs, quotation systems, invoices,
   memberships, automations, and AI assistants.
3. Operate as a Business Operating System rather than a website
   builder.
4. Support plugins, third-party integrations, white-label
   deployments, and multi-tenant SaaS.
5. Pass all Constitution quality gates with zero architectural debt.

Every PMM dimension reaches ≥ 95% before the platform is "complete."

## Amendment 4 — Platform Review Board + Health Report + Decision log (2026-07-05)

### Platform Review Board (PRB) — runs inside Phase 1 of every batch

Every batch begins with a **Platform Review Board (PRB)** — a
structured pre-implementation review. It is the substantive content
of Phase 1 (Architecture Review) and formalises the questions the
architect must answer before code is written.

The PRB answers, in order:

**Architecture**
- Is this still the best architecture?
- Can this subsystem be simplified?
- Is there unnecessary abstraction?
- Is there duplicated responsibility?
- Does this align with the Business OS?

**AI**
- Can this become more deterministic?
- Can metadata replace prompting?
- Can registries replace AI?
- Can configuration replace code?
- Can composition replace generation?

**Scale** — would this design still work with:
- 500 Apps
- 2,000 Sections
- 500 Containers
- 250 Layouts
- 100 Themes
- 50,000 Assets
- 100,000 Businesses
- Millions of generated pages

**Enterprise**
- Would an enterprise customer accept this?
- Would a Fortune 500 company build on it?
- Would another engineering team understand it?
- Would another AI model understand it?

**Performance** — estimate before implementation:
- memory
- rendering
- bundle size
- search speed
- registry lookup
- AI latency

**Developer Experience**
- Can another engineer understand this in six months?
- Can another AI continue development?
- Would onboarding a new developer take less than one day?

**Constitution scoring**
- Score the proposal against every Constitution category (§Registry
  Rules, §Design System, §Container System, §Component System,
  §AI Rules, §Quality Requirements, §Testing, §Performance,
  §Accessibility, §Code Standards, §Scalability Rule, §Loveable++
  Rule).
- If any category scores below 95%, explain why + whether the batch
  should proceed anyway.

### Platform Health Report (PHR) — runs at end of every milestone

At the end of every milestone, produce a Platform Health Report.
"Milestone complete" is not sufficient. The PHR must include:

- PMM score (each dimension + overall + delta from previous milestone)
- Technical debt inventory
- Architectural debt inventory
- Registry health (per-registry `.selfCheck()` output)
- Performance trend (vs prior milestone)
- Testing trend
- Accessibility trend
- Security trend
- AI quality trend
- Marketplace readiness
- Business OS completeness (per-layer scoring)
- Loveable++ comparison (per Loveable++ Rule category)
- Top 10 risks
- Top 10 opportunities
- Recommendation for the next milestone

The PHR is committed to disk under a `phr/` directory as
`phr/MN_health_report.md`, one per milestone.

### Decision recording (PLATFORM_DECISIONS.md)

Every significant architectural decision (proposal accepted, superior
design adopted, subsystem shape frozen, migration policy set, rule
adopted) is recorded in `PLATFORM_DECISIONS.md` as an Architecture
Decision Record (ADR).

Rules for the decision log:
- ADRs are **append-only**. Never edit historical ADRs; supersede
  them with a new entry citing the old.
- Every ADR must include: id, date, problem, options considered,
  chosen solution, why it was chosen, trade-offs, impact, migration
  required, Constitution references, related milestones.
- A superseded ADR is marked with `Status: SUPERSEDED by ADR-NNN`.
- ADRs are consulted by both humans and AI before any structural
  change to prevent unwittingly undoing deliberate choices.
- The M1–M2 decisions were retroactively seeded as ADR-001 through
  ADR-010 for continuity.

## Amendment 5 — Registry Governance Principle (2026-07-05)

### Registry Governance Principle (RGP)

Every registry — existing and future — must satisfy the RGP.
Violations block the next batch until resolved.

**RGP-1: Uniqueness of responsibility.**
No registry may duplicate another registry's responsibility. If two
registries would answer the same lookup, one must be a specialisation
of the other, and the relationship must be recorded in an ADR.

**RGP-2: Documented owner.**
Every registry file exposes a `REGISTRY_METADATA` constant declaring:
- `owner` — team or role responsible for evolution
- `purpose` — one-sentence responsibility statement
- `lifecycle` — one of `"alpha" | "beta" | "stable" | "deprecated"`
- `sinceVersion` — the semver of the platform when this registry
  first shipped
- `constitutionRefs` — array of Constitution section references
- `adrRefs` — array of ADR ids that constitute the registry

**RGP-3: Documented lifecycle.**
The `lifecycle` value governs consumer expectations:
- `alpha` — schema may change; not for production consumers
- `beta` — schema stable, contents growing, expect breaking changes
  only via ADR
- `stable` — schema + API frozen; only additive changes without ADR
- `deprecated` — sunset scheduled; new registrations forbidden

**RGP-4: Consistent API.**
Every registry facade must expose the following methods where
applicable to its domain (some are optional for registries with no
runtime query surface):

Required:
- `register(manifest)` — throws on duplicate; freezes on accept
- `get(id)` — returns undefined on miss
- `getOrThrow(id)` — throws on miss
- `has(id)` — boolean
- `list()` — all registrations
- `size()` — count
- `selfCheck()` — cross-registration invariants
- `snapshot()` — JSON-serialisable dump

Recommended:
- `search(query, limit?)` — weighted-keyword search
- `describe(id)` — AI-consumable one-liner
- `listByCategory(category)` — primary index
- `listByTag(tag)` — cross-cutting facet
- `resolveAlias(alias)` — rename support

Domain-specific:
- Any additional methods must be documented in the file header +
  covered by the registry's ADR.

**RGP-5: Health check protocol.**
At the end of every batch that touches registries, produce a
**Registry Health Report** with columns:

| Registry | Status | Complete | SelfCheck | Owner | PMM Impact |

- `Status` — lifecycle value
- `Complete` — percentage of expected content shipped
- `SelfCheck` — clean / warnings / errors
- `Owner` — from REGISTRY_METADATA
- `PMM Impact` — which PMM dimension this registry serves

The report is included in the Phase 5 Batch Report and appended to
the milestone Platform Health Report.

### Temporary Testing Exception — M3 B1–B4

**Recorded exception:** registry population tests are deferred to
Engineering Quality stream Week 5. Structural validation
(typecheck, lint, `selfCheck`, schema validation) remains
**mandatory from B1 onward**.

Justification: empty registries have no populated entries to test
beyond the kit's own guarantees, which are already covered by
`registryKit/__tests__/` unit tests (shipped in M1).

This exception is **time-boxed to M3 B1–B4**. From B5 onward, every
new registry ships with a minimum test suite (empty, duplicate,
freeze, alias, selfCheck).

## Amendment 6 — Registry Relationships + Canonical 18-Layer Model + Container Tier (2026-07-05)

### The canonical 18-layer Business OS model (authoritative)

Supersedes the informal layer list in Amendment 2. This is the
authoritative numbering used by every registry's
`REGISTRY_METADATA.relationships.businessOsLayer`.

| # | Layer | Description |
|---|---|---|
| 1 | Business | Trade / vertical / merchant profile (KG) |
| 2 | Brand | Name, logo, voice, visual identity, brand overrides |
| 3 | Theme | Complete design language — colours, motion, effect intensity |
| 4 | Design Token | Atomic values consumed by every component |
| 5 | Layout | Page-scale composition patterns |
| 6 | Navigation | How users move through the app |
| 7 | Container | Block-scale positioning primitives |
| 8 | Section | Content areas |
| 9 | Component | Atomic primitives (buttons, badges, inputs) |
| 10 | Forms | Field schemas + validation + submit adapters |
| 11 | Dashboard | Merchant Business OS control panel + tile blocks |
| 12 | Booking | Availability + calendar + slot + deposit + reminders |
| 13 | Marketplace | Apps + Packs + Templates + Plugins |
| 14 | Publishing | Publish pipeline + preview links + custom domain |
| 15 | AI | LLM tasks + provider routing + composition engine |
| 16 | Automation | Cron + workflows + growth coach + reminders |
| 17 | Enterprise | Teams + permissions + SDK + plugin loader + audit |
| 18 | Analytics | Warehouse dispatch + insights + reporting |

Runtime business systems (CRM, Payments, SEO, etc.) are Apps that
install into the Marketplace layer (13) and consume other layers.

### Registry Relationships (RGP-6 extension)

Every registry's `REGISTRY_METADATA` must now include a
`relationships` object declaring how it sits in the platform graph.
This is the foundation for AI reasoning, dependency validation,
plugin loading, lazy loading, marketplace packaging, and enterprise
scalability.

**Required `relationships` shape:**

```ts
type RegistryRelationships = {
  /** Business OS layer (1..18) this registry backs. */
  businessOsLayer: number;

  /** Registries this registry reads from at register() or lookup time.
   *  Empty for root registries (e.g. designTokenRegistry). */
  upstreamDependencies: readonly string[];

  /** Registries known to consume this registry's entries. Populated
   *  progressively as new registries land. */
  downstreamDependents: readonly string[];

  /** Composition role in the platform graph:
   *  - "root"          — no upstream; has downstream (foundation)
   *  - "intermediate"  — both upstream and downstream (most)
   *  - "leaf"          — has upstream; no downstream (terminal)
   *  - "standalone"    — no upstream, no downstream (rare; utility) */
  composition: "root" | "intermediate" | "leaf" | "standalone";

  /** Whether third-party plugins may register into this registry
   *  post-boot. Governs marketplace + plugin system compatibility. */
  pluginCompatible: boolean;

  /** Optional: if this registry is planned to become a specialisation
   *  of a broader registry, name the future parent. Preserves the
   *  migration path in the metadata itself. */
  futureParent?: string;
};
```

### Container Tier (RGP-7)

Every registration in `designSystemRegistry` with
`category === "containers"` must declare an explicit tier field:

```ts
tier: "layout" | "content" | "utility"
```

Semantics:
- **layout** — page-scale root container. Exactly one per rendered
  page. Owns page-scale rhythm.
- **content** — block-scale container that slots inside a layout
  container and holds sections or nested content containers.
- **utility** — positioning wrapper (overlay, sticky, floating) that
  wraps content without owning content itself.

Layout Engine (M3 B4) will validate at register time:
- Every layout must have exactly one layout-tier root.
- Content-tier containers slot only inside layout or content
  containers.
- Utility-tier containers wrap without becoming a slot themselves.

### Registry Health Report — updated columns

Per RGP-5 (Amendment 5), extended for Amendment 6:

| Registry | Layer | Status | Complete | SelfCheck | Owner | Composition | Plugin? | PMM Impact |

New columns:
- **Layer** — Business OS layer number
- **Composition** — root / intermediate / leaf / standalone
- **Plugin?** — pluginCompatible boolean

The report is included in every Phase 5 Batch Report and every
milestone Platform Health Report.

### Migration policy for RGP-6 backfill

All 15 existing registries must have `relationships` metadata declared
before M3 exits. Backfill scope is tracked as its own batch (M3 B2.5
per approved plan).

## Amendment 7 — Business Growth Intelligence layer (2026-07-05)

### The intelligence architecture

The platform's decision intelligence sits above the Foundation and
below the Composition layers. It answers *what should this specific
business do* — not just *how to render a page*.

**Data (5 registries — Business OS layer 1):**
1. `businessProfileRegistry` — WHO the merchant is (stable identity)
2. `growthStrategyRegistry` — WHAT they want now (dynamic quarterly)
3. `playbookRegistry` — reusable single-concern patterns with typed
   facets (renamed from Conversion Playbooks — will span SEO, brand,
   trust, accessibility, performance, etc.)
4. `websiteRecipeRegistry` — lightweight orchestrators that compose
   playbooks (no duplicated decision logic)
5. `facetKindRegistry` — schema + merge rules per facet kind
   (extensibility mechanism — new subsystems register their own kinds
   without touching playbook code)

**Logic (1 service):**
- `strategyResolver` — pure business logic. Loads playbooks by
  recipe, applies merge strategies per facet kind, validates, freezes.
  Returns a typed `ResolvedStrategy`. Every consumer (Composer,
  Dashboard, Marketing, SEO, Booking, CRM) uses the same resolver.

**Assembler (many):**
- Composer, dashboard builder, marketing builder, etc. — each
  consumes `ResolvedStrategy` and assembles its output. Zero merge
  logic. Ever.

### Playbook facet model (RGP-8)

Playbooks contain a **typed facet tree**. Not a string map at the API
surface — the internal registry uses dotted `facetKind` ids
(`gallery.style`, `pricing.display`) but ResolvedStrategy exposes a
structured object:

```
resolved.gallery.style          // "before-after"
resolved.pricing.display        // "package"
resolved.cta.primary            // { label, href }
resolved.seo.pageKinds          // ["town", "county"]
```

**Playbooks own intent, never copy.** A playbook says
`hero.messageStrategy = "primary-service-emphasis"` — the actual
headline text comes from the AI content layer + KG package.

### Evidence dimensions (RGP-9)

Every playbook + recipe declares FIVE evidence dimensions:

- `confidence: 0-100` — author-declared belief
- `evidenceStrength: "none" | "anecdotal" | "measured" | "validated"`
- `sampleSize: number` — websites / merchants observed
- `lastValidated?: ISO 8601` — when confidence was refreshed
- `marketsValidated?: string[]` — which countries the evidence holds in

M6 `evidenceRegistry` + `businessOutcomeRegistry` will populate + weight
these automatically. Forward-compatible from B4.5.

### Deferred to M5/M6 (documented, not shipped)

- `merchantPreferencesRegistry` — merchant *preferences* (dark theme,
  never-show-pricing, always-WhatsApp) separated from Business
  Profile. Different concept, different persistence.
- `businessCapabilityRegistry` — structured capabilities per trade
  (each capability declares required photos, pricing model, FAQs,
  compliance, warranties, upsells, average duration, seasonal demand).
  Replaces string-typed `services[]` in KG packages.
- `evidenceRegistry` + `businessOutcomeRegistry` — empirical
  performance data feeding automatic playbook ranking.

### Composer contract (RGP-10)

The composer is authorised to use exactly ONE LLM call per
generation (Step 1 — intent classification). Every other step is
deterministic. Merging strategy data inside the composer is
explicitly forbidden — that responsibility belongs to the
`strategyResolver`.
